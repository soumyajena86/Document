STEPS to build Provisioning
-----------------------------
Eclipse setup
-------------
1.Download java 1.8 and add it under windows/preference
2.Downlod maven version>3.6.2 and select it under windows/preference/maven/installation 
then

1. Checkout the code from github - https://github.cerner.com/millennium-security/security-provisioning
2. Import the code as general project in eclipse
3. Change the version of certain jars in pom.xml
4. If Any java project not showing as java project then right click on the project and configure/convert to faceted form
5. Select individual project,right click and select Maven/update project to add all the dependancy
6. Then select root pom i.e. pom.xml inside security-provisioning and do run as/run config/maven and give the goal as clean install -Dmaven.buildNumber.skip=true and select the skip test check box and then run
7. The above process will generate ear and war

Running Security Provisioning on Alva  (https://github.cerner.com/millennium-security/security-provisioning/blob/master/security-provisioning-alva/running.md)
---------------------------------------
1. Install docker (To run and debug the application in Docker)
2. Run below commands from powershell where the directory should point to security-provisioning/target folder
# docker rmi any images
# docker rm any containers
# create target directory
rm -r target
mkdir target
cd target
# Make sure we have the latest jar 
wget -O alva-lonewolf.jar "https://cernerrepos.net/maven/com/cerner/alva/alva-lonewolf/0.80/alva-lonewolf-0.80.jar"
# Make sure lonewolf_ml-security-provisioning folder is empty (it gets generated in the command after this)
rm -r lonewolf_ml-security-provisioning
# If using a local copy of the service profile, replace 'ml-security-provisioning' with the path to your local service profile (../ml-security-provisioning.yaml)
java -jar alva-lonewolf.jar ml-security-provisioning |& tee alva-lonewolf-provisioning.log
# By default, the command above will generate a docker image for DEVENG. For other domains, you will need to specify the tenant:
java -jar alva-lonewolf.jar ml-security-provisioning -tenant msol.ip.devcerner.net 
3. Then do changes in the config.properties as present in this folder.
4. To startup the provisioning on Alva docker container run below command
cd lonewolf_ml-security-provisioning/
docker-compose up -d
5.To see the logs: docker logs lone_wolf |& tee ../lone_wolf.log or use docker logs lone_wolf -f



java -jar alva-lonewolf.jar ./ml-security-provisioning.yaml -tenant msol.ip.devcerner.net

We use symmetric key encryption technique for encryption/decryption of the targetId(combination of millenium_domain-name i.e millennium_a3d85d2b78963ca6):

Encryption :

1. Create keystore
2. Store all the details as secret Key by concatinating each other by "%" symbol like (Enter Client Mnemonic:,Enter Key Version:,currentDate,expirationDate) in keystore as provisioningKey alias
3. Ask for domain name e.g. deveng/msol/eng18
4. Encrypt the domain name by using algorithim(PBEWithHmacSHA1AndAES_128 ) by using secret key (got as part of step-2)
5. The target Id will be like "millennium_a3d85d2b78963ca"

Decryption :

1. 
1. Read the secret key from the keystore using alias
2. Use decryption algorithim(PBEWithMD5AndDES) which uses secret key (received as part of previous step)
3. Get the domain name as part of decryption
4. match the domain name with the defined domain name as part of Environment variable i.e. "millennium/domainName"
5. Wrong target Id will throw "Authentication Error: Invalid targetID"

Brfore any opration targetID is validated which has been passed as part of request. If its a valid target id then the current  operation is allowed or else the application throws an exception.

The following are the "supported" account states that are generally used in Millennium.

Active - The personnel record exists, is active (ACTIVE_IND=1), and effective. The corresponding username exists in the authorization database and does not have the "Disabled" restriction.
Inactive - The personnel record exists but is inactive (ACTIVE_IND=0). The corresponding username does not exist in the authorization database.
Deleted - Same as Inactive.
Suspended - The personnel record exists. The corresponding username exists in the authorization database and has the "Disabled" restriction.
Disabled - Same as Suspended.
 

In the context of the provisioning adapter, the suspend operation would be equivalent to the suspended status in Millennium. The PRSNL table record exists, but is set to END_EFFECTIVE_DT_TM of the moment of suspension. The EA_USER table row exists, but the "Disabled" restriction is applied to the user account so it cannot log in.

 

(details on the Disabled restriction: https://wiki.ucern.com/display/public/reference/AuthView+Uti...)

 

The "Delete" operation is the same as Inactive in Millennium. The personnel record exists, and the USERNAME field is still populated with the username that was used, but is marked as ACTIVE_IND=0 and the END_EFFECTIVE_DT_TM is set to the moment of suspension. The EA_USER table row is deleted.




The Inactivate option corresponds to the Enterprise Provisioning Service Delete operation.

When you change the status of a personnel member (activate or inactivate), the following actions occur with an associated user account:

When a personnel member is inactivated, their associated user account (if any) is removed from the Cerner Millennium user directory.
When an inactive personnel member is activated, a user account is created for the personnel (assuming the personnel has a username), setting the password to the username.
When a personnel member is suspended, their associated user account (if any) is disabled.
When a suspended personnel member is renewed, their associated user account (if any) is re-enabled.

Weblogic Migration:
--------------------
1.To verify the WLS-compatible EAR/WAR update the system-enterprise-appliance-verifier-plugin to version 2.0.0.
2.The POM files have to be updated to generate two versions of EARs - one for the IBM WAS and the other one for the Oracle WebLogic Server
3.system-jdbc-javaee-ml version must be latest [2.0.1]
4.system-registry-genesis-javaee-ml must have latest version [2.0.0]
*5.exodus-instrument-servlet dependency must have latest version [3.0.1]
6.com.cerner.mmf.cps.system.runtime.j2ee.J2EE25RuntimeListener must be added as the first Listener. 
When mmf-cps-system-runtime-j2ee jar dependency is added to the pom.xml, adding below listener to web.xml (as very first listener) is mandatory.

pom.xml changes [Dependency changes]
1.Update artifact id from *-ear to *-wls-ear
2.System Runtime Jar
3.Replace system-runtime-websphere with mmf-cps-system-runtime-j2ee jar.
<dependency>
   <groupId>com.cerner.mmf.cps</groupId>
   <artifactId>mmf-cps-system-runtime-j2ee</artifactId>
   <version>{latest.version}</version>
   <scope>runtime</scope>
</dependency>
4.mmf-cps-system-runtime-j2ee dependency transitively brings in the below dependencies, If the below transitive dependencies are already defined in your pom.xml , update them to the latest available version.

com.cerner.system:system-core:jar
com.cerner.system:system-j2ee:jar
javax.servlet:servlet-api:jar

5. Add/update system-context.jar as a dependency- to avoid java.lang.NoClassDefFoundError: com/cerner/system/context/ContextManager caused by system-registry-genesis-javaee-ml dependency
<dependency>
    <groupId>com.cerner.system</groupId>
    <artifactId>system-context</artifactId>
    <version>{latest.version}</version>
    <scope>runtime</scope>
</dependency>
6.The mmf-cps-system-runtime-j2ee jar has an implementation of ServeletContextListener i.e. J2EE25Listener. When this dependency is added to the pom.xml, adding below listener to web.xml (as very first listener) is mandatory. This listener gets notified by the container when the context is initialized(on server startup)/destroyed(on server shutdown). This is responsible for unregistering MBeans associated with the application instance from the MbeanServer and shuts down the executor service thus cleaning up system resources.
<listener>
    <display-name>J2EE Context Listener for Runtime APIs</display-name>
    <listener-class>com.cerner.mmf.cps.system.runtime.j2ee.J2EE25RuntimeListener</listener-class>
</listener>

        this.value = value;
    }

        return value;
    }

protected SiQueryResponse executeQueryProto(
      SiQuery siQuery, String tenant, AuthorizationContext authorizationContext, String siUrl) {

    log.info("Building SIClient to run the provided SI query");
    SIClient siClient = getSIClient(siUrl);

    try (InputStream byteArrayInputStream = new ByteArrayInputStream(siQuery.toByteArray())) {
      ExecuteQueryProtoRequest.Builder builder = ExecuteQueryProtoRequest.builder();
      builder.query(byteArrayInputStream).tenantId(tenant);
      if (ObjectUtils.isNotEmpty(authorizationContext)) {
        byte[] authZContextInBytes = authorizationContext.toProto().toByteArray();
        String authZEncodedContext = Base64.getUrlEncoder().encodeToString(authZContextInBytes);
        builder.ehrcDataAuthzContext(authZEncodedContext);
      }
      ExecuteQueryProtoRequest executeQueryProtoRequest = builder.build();

      log.info("Executing executeQueryProtoRequest");
      ExecuteQueryProtoResponse executeQueryProtoResponse =
          siClient.executeQueryProto(executeQueryProtoRequest);

      InputStream inputStream = executeQueryProtoResponse.getInputStream();
      log.info(
          "ExecuteQueryProto response code: {}", executeQueryProtoResponse.get__httpStatusCode__());

      return SiQueryResponse.parseFrom(inputStream);
    } catch (BmcException e) {
      log.error("ExecuteQueryProto request failed with status code {}", e.getStatusCode(), e);
      throw new RuntimeException(e);
    } catch (Exception e) {
      log.error("ExecuteQueryProto request failed", e);
      throw new RuntimeException(e);
    }
  }




protected SIClient getSIClient(String endpoint) {
    final InstancePrincipalsAuthenticationDetailsProvider provider =
        InstancePrincipalsAuthenticationDetailsProvider.builder()
            .metadataBaseUrl(INSTANCE_PRINCIPAL_METADATA_BASE_URL)
            .build();

    return SIClient.builder()
        .endpoint(endpoint)
        .configuration(
            ClientConfiguration.builder()
                .connectionTimeoutMillis(Integer.MAX_VALUE)
                .readTimeoutMillis(Integer.MAX_VALUE)
                .retryConfiguration(RetryConfiguration.NO_RETRY_CONFIGURATION)
                .build())
        .build(provider);
  }



protected SiQuery buildSqlParameterizedQuery(String sqlQuery, Map<String, Value> datamap) {
    return SiQuery.newBuilder()
        .setQueryProviderType(SiQuery.QueryProviderType.ORACLE)
        .setQueryProvider(
            SiQuery.QueryProvider.newBuilder()
                .setOracleQueryProvider(
                    SiQuery.QueryProvider.OracleQueryProvider.newBuilder()
                        .setSql(
                            SiQuery.SQL
                                .newBuilder()
                                .setQuery(sqlQuery)
                                .putAllNamedPlaceHolderData(datamap)
                                .build())
                        .build())
                .build())
        .build();
  }

To store the SSH public key in OCI Vault:

1. Log in to OCI console and navigate to Identity & Security > Vault.
2. Create a new vault or use an existing one.
3. Go to the 'Secrets' tab and click 'Create Secret'.
4. Select 'Plaintext' and enter a name (e.g., 'ssh-public-key').
5. Paste the contents of your `id_rsa.pub` file.
6. Click 'Create Secret'.

After creating the secret, grant necessary permissions to the dynamic group or service account used by your Terraform configuration to read the secret.

Example policy: Allow dynamic-group to read secret-family in compartment


You can store the SSH public key in OCI Vault and retrieve it in your Terraform configuration. To do this:

1. Create a secret in OCI Vault for the SSH public key.
2. Use the OCI Terraform provider to retrieve the secret value.


Example:
data "oci_vault_secret" "ssh_public_key" {
  secret_id = "ocid1.vaultsecret.oc1..<your_secret_ocid>"
}

locals {
  ssh_authorized_key = data.oci_vault_secret.ssh_public_key.secret_bundle.secret_ocsp_response_content
}

Then, use local.ssh_authorized_key in your appliance.tf as before.

This approach securely stores and manages your SSH public key in OCI Vault.

To get the `key_id`, you'll need to create a Key in the OCI Key Management Service (KMS). Here's how to do it:

1. __Create a Key__: You can create a Key using the OCI console or Terraform.

Using Terraform, you can create a Key like this:

```terraform
resource "oci_kms_key" "example_key" {
  compartment_id = "ocid1.compartment.oc1..<your_compartment_ocid>"
  display_name   = "example-key"
  key_shape {
    algorithm = "AES"
    length    = 32
  }
}
```

After creating the Key, you can retrieve its OCID using the `oci_kms_key` data source.

```terraform
data "oci_kms_key" "example_key" {
  key_id      = oci_kms_key.example_key.id
  management_endpoint = oci_kms_vault.example_vault.management_endpoint
}
```

The `id` attribute of the `oci_kms_key` resource or data source will give you the `key_id`.

To use the Secret Manager in OCI with Terraform, you'll need to use the OCI Terraform provider. Here's a step-by-step guide:

1. __Create a Vault__: First, create a Vault in OCI using Terraform. You'll need to specify the compartment ID and the vault type.

```terraform
resource "oci_kms_vault" "example_vault" {
  compartment_id = "ocid1.compartment.oc1..<your_compartment_ocid>"
  display_name   = "example-vault"
  vault_type     = "DEFAULT"
}
```

2. __Create a Secret__: Once the Vault is created, you can create a secret using Terraform. You'll need to specify the vault ID, key ID, and the secret content.

```terraform
resource "oci_vault_secret" "example_secret" {
  compartment_id = "ocid1.compartment.oc1..<your_compartment_ocid>"
  vault_id       = oci_kms_vault.example_vault.id
  key_id         = "ocid1.key.oc1..<your_key_ocid>"
  secret_content {
    content_type = "PLAINTEXT"
    content      = "your_secret_content"
  }
  secret_name = "example-secret"
  description = "Example secret"
}
```

3. __Retrieve the Secret__: To retrieve the secret in your Terraform configuration, you can use the `oci_vault_secret` data source.

```terraform
data "oci_vault_secret" "example_secret" {
  secret_id = oci_vault_secret.example_secret.id
}
```

4. __Use the Secret__: You can then use the retrieved secret in your Terraform configuration.

```terraform
locals {
  secret_content = data.oci_vault_secret.example_secret.secret_bundle.secret_ocsp_response_content
}
```

STORY 1: Unified Provisioning Platform (BFF Architecture)
🔷 System Design Story (How to present)
Problem

We had a highly manual provisioning system where users had to interact with 15+ different systems, each with its own authentication mechanism.
Provisioning took ~220 minutes per user, impacting a large-scale rollout (~$23M cost impact).

High-Level Design

I designed a BFF-based architecture with a connector framework:

UI → BFF → Auth Broker → Connectors → 15 Systems
Components:
BFF Layer
Handles session, orchestration, aggregation
Auth Broker
Token exchange, credential management
Connector Layer
One adapter per system (OIDC, SAML, API key, etc.)
Vault
Secure storage for secrets
Audit + Logging
Observability and compliance
Key Design Decisions
Avoid exposing tokens to frontend → security
Use pluggable connectors → scalability for new systems
Introduce bulk provisioning (spreadsheet ingestion) → operational efficiency
Design for idempotency and retry → reliability
Challenges
Handling 15 different auth mechanisms
Ensuring consistency across distributed systems
Managing partial failures
Maintaining backward compatibility
Trade-offs
Added backend complexity vs simpler UI
Slight latency increase vs improved security and maintainability
Impact
Reduced provisioning time by 45% (220 → 122 mins)
Roadmap to ~25 mins
Enabled batch provisioning
Improved scalability and maintainability
⭐ STAR Answer (Behavioral)

S (Situation):
User provisioning across 15 systems was manual and time-consuming (~220 mins/user).

T (Task):
Design a scalable architecture to unify provisioning and reduce effort.

A (Action):

Designed BFF architecture with connector pattern
Introduced auth broker for token mediation
Built bulk provisioning feature
Led implementation and team delivery

R (Result):

Reduced provisioning time by 45%
Enabled automation roadmap to ~25 mins
Improved system scalability and reduced operational cost

Standard schema across systems
Challenges

Integrating with policy engine (SOM + SI)
Impact
Unified audit across platform

Enabled real-time policy validation
